# 박람회(행사) 예약 관리 플랫폼 — 개발 기획서

> 참여기업 심화 프로젝트 주제 1번(박람회 예약 관리 플랫폼)을 **교육용 경량 범위**로 재구성한 기획서다.
> 원 명세의 대규모 요소(다중 테넌트 SaaS, MongoDB, React SPA, 정산 자동화 등)는 제외하고,
> **3개 역할의 핵심 흐름 + LMS(교육 이수)** 를 구현 대상으로 확정한다.
> 상세 기능 정의: [기능명세서.md](기능명세서.md)

## 1. 프로젝트 개요
- **한 줄 정의**: 전체 관리자가 여러 박람회를 개설하고, 박람회 관리자가 자기 행사를 운영하며, 고객이 모바일웹에서 예약·결제·QR 입장하는 예약 관리 플랫폼.
- **핵심 가치**: ① 흩어진 수기 예약을 한 곳으로 ② 결제~QR 입장까지 온라인 완결 ③ 운영자 교육(LMS)까지 내장.
- **차별 포인트(가산점 요소)**: 실제 카카오페이(PortOne) 결제, QR 전자 티켓, YouTube 기반 교육 이수율 자동 집계.

### 1.1 경량화 방침
- 원 명세의 "SaaS 다중 테넌트/광고 과금/자동 정산/다국어" 등은 **개념만 반영**하고 과금·정산 로직은 만들지 않는다.
- 프론트는 React가 아니라 **Thymeleaf + jQuery/Ajax** (공통 스택), DB는 **H2 파일 모드** 단일 사용(MongoDB 미사용).
- "구현이 쉽고 결과물이 눈에 보이는" 기능(목록/상세/예약/결제/QR/대시보드/교육 영상) 위주로 선택한다.

## 2. 제출물 요건
- 3개 역할(전체 관리자·박람회 관리자·고객)이 각자 화면에서 동작하는 것을 시연 영상으로 증빙.
- 결제(카카오페이) 성공 → 예약 확정 → QR 발급 → 현장 체크인까지 한 흐름 시연.
- LMS: 교육 생성 → 박람회 관리자 수강(영상 끝까지) → 이수 완료 → 전체 관리자 수강율 확인.

## 3. 기술 스택 (확정)
- **Java 17 / Spring Boot 3.x / Gradle** (공통 규칙 준수 · Boot 4·Java 25로 올리지 않음)
- **Spring Data JPA** (MyBatis 미사용), **H2 파일 모드**(추후 MySQL 전환은 application.yml 교체만)
- **Thymeleaf + jQuery/Ajax**, 서버 렌더링
- **Spring Security** (전체/박람회 관리자, 세션·역할 기반) + **경량 세션 인증**(고객: 전화번호+PIN)
- 결제: **PortOne V2 (카카오페이)**, 키 미설정 시 **Mock 결제 폴백**
- QR: 서버에서 QR 이미지 생성(경량 라이브러리 또는 데이터 URI)
- 교육 영상: **YouTube IFrame Player API** 로 재생·진행률 집계

### 3.1 채택하지 않은 명세 항목 (보고서 근거)
| 명세 항목 | 처리 | 사유 |
|---|---|---|
| React SPA / MyBatis 병행 | 미채택 | 공통 스택(Thymeleaf/JPA)으로 대체 |
| MongoDB(감정분석/비정형) | 제외 | 교육용 범위 밖, 단일 RDB로 충분 |
| 회계팀·마케팅/스폰서 역할 | 제외 | 원 명세에서 **(확장)** 으로 표기된 역할 |
| 자동 정산·수수료·송금 리포트 | 제외 | 결제 성공 표시까지만 |
| VIP 배너 **과금/노출 추적** | 제외 | 배너 **노출·우선순위**까지만(광고 집행 로직 없음) |
| QR **카메라 스캐너** 모드 | 대체 | 카메라 대신 **코드 입력 기반 체크인** |
| 다국어·PG 다중연동·이메일/SMS 템플릿 | 제외 | 범위 최소화 |
| Jenkins/Docker/K8s/모니터링 | 제외 | 로컬 실행·시연 중심 |

## 4. 기능 범위 (명세 대비 구현 계획)

### 4.1 공통
- 관리자 로그인/로그아웃(Spring Security), 고객 로그인(전화번호+PIN 경량 세션).
- 역할 기반 접근 제어: 전체 관리자 / 박람회 관리자 / 고객.

### 4.2 전체 관리자 (Super Admin)
- 박람회 CRUD(개설/수정/삭제) + **행사 식별코드 자동 발급**.
- 박람회 관리자 **계정 발급/비활성** + 담당 박람회 지정.
- **VIP 배너** 등록(이미지·링크·노출순위·기간) — 과금 없음.
- **전체 대시보드**(총 박람회/총 예약/총 매출/평균 체크인율) + 통합 통계.
- **LMS**: 교육 생성(유튜브 링크) · 대상(박람회 관리자) 배정 · **수강율 조회**.

### 4.3 박람회 관리자 (Event Admin)
- 자기 박람회 **대시보드**(예약 수·결제 금액·체크인율).
- **예약자 명단**(검색/필터/상태 변경/**CSV 다운로드**).
- **체크인 처리**(예약번호/체크인 코드 입력, QR 대조).
- **콘텐츠·배너 편집**(소개문·일정·가격·좌석·썸네일).
- **교육 수강**(배정된 영상 시청 → 시청 진행률 자동 기록 → 이수).

### 4.4 고객 (User)
- **행사 목록**(검색: 카테고리/날짜/지역) + **VIP 배너 노출**.
- **행사 상세**(소개·일정·장소·가격·**남은 좌석**).
- **예약 신청**(인원·연락처·동의) → 예약 레코드 생성(결제대기).
- **결제**(카카오페이/PortOne, Mock 폴백) → 예약 확정 + **QR 티켓 발급**.
- **나의 예약**(예약 내역·QR 재조회·취소 요청).

### 4.5 LMS (교육 이수) — 추가 기능
- 전체 관리자가 교육(제목·설명·YouTube 링크)을 만들고 박람회 관리자에게 배정.
- 박람회 관리자는 배정된 교육을 시청. **YouTube IFrame API**로 재생 위치를 주기 집계하여 **시청률(%)** 을 서버에 저장.
- **영상 끝(길이의 95% 이상)** 에 도달하면 **이수 완료** 로 전환.
- 전체 관리자는 대상자별 **수강율/이수 여부** 를 조회한다. (화면 톤 = 첨부 스크린샷: 이수 현황·예상 소요시간·이수 혜택·영상 플레이어·시청률 바)

## 5. 데이터 모델 (ERD 초안)
```
AdminUser (전체/박람회 관리자)
  id, username(uniq), password(BCrypt), name, role[SUPER_ADMIN|EVENT_ADMIN],
  active, managed_expo_id → Expo(nullable, EVENT_ADMIN만), created_at

Expo (박람회)
  id, code(uniq, 자동발급), title, description, category, venue,
  start_at, end_at, price, capacity, status[DRAFT|OPEN|CLOSED|HIDDEN],
  thumbnail_url, created_at

Customer (고객)
  id, name, phone(uniq), pin(BCrypt), created_at

Reservation (예약)
  id, expo_id → Expo, customer_id → Customer, headcount, contact_phone, amount,
  status[PENDING|CONFIRMED|CHECKED_IN|CANCEL_REQUESTED|CANCELLED],
  reservation_no(uniq), checkin_token(uniq), paid_at, checked_in_at, created_at

Payment (결제)
  id, reservation_id → Reservation, provider[PORTONE|MOCK], payment_id,
  amount, status[PAID|FAILED|CANCELLED], paid_at

Banner (VIP 배너)
  id, title, image_url, link_url, priority, active, start_at, end_at

Training (교육)
  id, title, description, youtube_url, youtube_video_id, active,
  created_by → AdminUser, created_at

TrainingAssignment (교육 배정/수강)
  id, training_id → Training, assignee_id → AdminUser,
  status[NOT_STARTED|IN_PROGRESS|COMPLETED],
  progress_rate(double, %), last_position_sec, duration_sec, completed_at, created_at
```
**관계 요약**: Expo 1—N Reservation, Customer 1—N Reservation, Reservation 1—N Payment(재시도 대비, 최종 PAID 채택),
AdminUser N—1 Expo(담당), Training 1—N TrainingAssignment, AdminUser 1—N TrainingAssignment(수강자).
**N+1 회피**: 예약 목록은 `Expo`·`Customer` fetch join, 배정 목록은 `Training`·`AdminUser` fetch join.

## 8. 개발 단계 (마일스톤)
1. **M1 기반**: 프로젝트 생성, Security(관리자)·고객 세션, 시드 데이터(관리자/박람회/고객 샘플).
2. **M2 박람회·예약**: Expo CRUD(코드 발급), 고객 목록/상세/예약, 좌석 잔여 검증.
3. **M3 결제·QR**: PortOne(카카오페이) 결제·서버검증·Mock 폴백, 예약 확정, QR 발급, 체크인.
4. **M4 관리자**: 박람회 관리자 명단/상태변경/CSV, 콘텐츠·배너 편집, 대시보드/통계.
5. **M5 LMS**: 교육 생성·배정, YouTube 수강 진행률 집계, 이수 처리, 수강율 조회.
6. **M6 마감**: 배너 노출, 예외/검증 정리, 시연 시나리오 리허설.

## 9. 시연 시나리오 (영상용 초안)
1. 전체 관리자 로그인 → 박람회 개설(코드 자동 발급) → 박람회 관리자 계정 발급.
2. 고객 로그인 → 행사 목록/상세 → 예약 → **카카오페이 결제** → 예약 확정·**QR 발급**.
3. 박람회 관리자 로그인 → 예약자 명단 확인 → **체크인 코드로 입장 처리** → 대시보드 지표 변화.
4. 전체 관리자 → **교육 생성(유튜브 링크)** → 박람회 관리자에게 배정.
5. 박람회 관리자 → 교육 화면에서 **영상 끝까지 시청** → **이수 완료**.
6. 전체 관리자 → **수강율 화면**에서 이수 여부 확인.

## 13. 데이터 모델 설계 검토 (spring-data-jpa 스킬 반영)
> `spring-data-jpa` 스킬 지침(애그리거트 경계·관계·N+1·프로젝션)으로 §5 모델을 점검한 결과다. (스킬은 Boot 4/Java 25 기준이라 문법은 Boot 3.x/Java 17에 맞게 적용)

### 13.1 저장소(Repository) 경계 — 애그리거트 루트만
- 저장소 생성: **AdminUser, Expo, Customer, Reservation, Banner, Training**.
- **Payment는 Reservation 애그리거트의 하위 엔티티**로 두고 별도 저장소를 만들지 않는다(cascade 저장). **매출 KPI는 Payment가 아니라 Reservation(status ∈ CONFIRMED·CHECKED_IN)의 amount 합**으로 산출한다.
- **TrainingAssignment**는 조인 엔티티지만 (수강자별·교육별) 양방향 조회가 필요해 **예외적으로 저장소를 둔다**(읽기 중심).

### 13.2 관계 — @ManyToOne(LAZY) 우선, @OneToMany 지양
- 모든 참조는 `@ManyToOne(fetch = LAZY)`: Reservation→Expo/Customer, AdminUser→Expo(담당), Payment→Reservation, TrainingAssignment→Training/AdminUser.
- **Expo·Customer·Training·AdminUser에 `@OneToMany` 컬렉션을 두지 않는다.** 목록은 항상 다(many) 쪽 저장소(Reservation/TrainingAssignment)에서 조회한다(대량 로딩·N+1 예방).
- `@ManyToMany` 미사용 — 교육↔관리자는 조인 엔티티(TrainingAssignment)로 명시.

### 13.3 N+1 · 읽기 성능
- 예약자 명단: `JOIN FETCH r.customer`(필요 시 expo)로 조회.
- 수강율 조회·대시보드·통계: **DTO 프로젝션 + COUNT/SUM 집계 쿼리**, 조회 서비스는 `@Transactional(readOnly = true)`. 엔티티를 화면까지 끌고 가지 않는다.

### 13.4 확인이 필요한 리스크
- **좌석 오버셀(동시성)**: 잔여 = capacity − Σheadcount(CONFIRMED·CHECKED_IN). **PENDING은 좌석을 점유하지 않으므로**(결제 성공 시에만 확정) 미결제 예약 만료 로직이 불필요하다. 오버셀은 **결제 확정 트랜잭션 안에서 잔여를 재검증(필수)** 해 막고, 동시 폭주 대비로 `Expo`에 `@Version`(낙관적 락) 적용은 권장(선택).
- **결제 멱등성**: PortOne 콜백 재진입 시 중복 확정 방지 — `Payment.paymentId` 유니크 + 이미 CONFIRMED면 무시.
- **수강율 부풀리기 방지**: 클라이언트 폴링 **5초 주기**, 서버는 **최대 도달 위치가 증가할 때만 UPDATE**(되감기·시킹 무시). 시청 이벤트 로그 테이블은 만들지 않고 progressRate/lastPositionSec만 비정규화 저장.
