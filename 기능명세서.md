# 박람회(행사) 예약 관리 플랫폼 — 기능명세서

> 개발 기획서([개발기획서.md](개발기획서.md))에서 확정한 범위의 **기능 단위 상세 명세**다.
> 각 기능에 식별자(F-CODE)를 부여하고 입력/출력·상태·권한·검증을 정의한다.

## 1. 문서 개요
### 1.1 목적
- 3개 역할(전체 관리자·박람회 관리자·고객)과 LMS 기능을 개발·검증 가능한 단위로 명세한다.

### 1.2 범위
- 포함: 인증, 박람회 관리, 예약·결제(카카오페이)·QR·체크인, 대시보드/통계, VIP 배너, LMS(교육).
- 제외: 자동 정산·수수료, 광고 과금·추적, QR 카메라 스캐너, 다국어, MongoDB, 회계/마케팅 역할.

### 1.3 용어 정의
| 용어 | 의미 |
|---|---|
| 박람회(Expo) | 전체 관리자가 개설하는 개별 행사(코드로 식별) |
| 예약(Reservation) | 고객이 특정 박람회에 대해 생성한 참가 신청 |
| 체크인 코드 | 예약 확정 시 발급되는 입장 확인용 토큰(QR payload) |
| 이수 | 교육 영상을 끝까지 시청해 수강이 완료된 상태 |
| 시청률 | (시청한 최대 위치 ÷ 영상 길이) × 100 (%) |
| 이수 임계 | 시청 최대 위치 ≥ 영상 길이의 95% (COMPLETED 전환 기준 상수) |

## 2. 사용자 역할 및 권한
### 2.1 역할
- **전체 관리자(SUPER_ADMIN)**: 플랫폼 전역. 박람회/계정/배너/통계/교육 관리.
- **박람회 관리자(EVENT_ADMIN)**: 담당 박람회 1건 운영. 예약/체크인/콘텐츠, 배정 교육 수강.
- **고객(USER)**: 행사 탐색·예약·결제·QR·나의 예약. (전화번호+PIN 경량 세션)

## 4. 기능 상세 명세

### 4.1 인증·계정
#### F-AUTH-01 관리자 로그인/로그아웃
- 입력: username, password. 출력: 세션(역할별 랜딩). 저장: BCrypt 대조.
- 검증: 실패 시 오류 메시지. `active=false` 계정 로그인 차단.
- 권한: SUPER_ADMIN → 전체 콘솔, EVENT_ADMIN → 담당 박람회 콘솔.

#### F-AUTH-02 고객 로그인/식별
- 입력: 전화번호 + PIN(4~6자리). 출력: 고객 세션(CustomerSession).
- 최초 전화번호면 가입(이름·PIN 설정), 이후 로그인. PIN은 BCrypt 저장.

#### F-ACCT-01 박람회 관리자 계정 발급/비활성
- 전체 관리자가 아이디/초기 비밀번호 생성 + 담당 박람회 지정. 출력: 자격증명.
- 비활성 토글(`active`)로 로그인 차단(계정 삭제 대신 보존).

### 4.2 박람회·배너
#### F-EXPO-01 박람회 CRUD + 코드 자동발급
- 입력: 제목·설명·카테고리·장소·일정(시작/종료)·가격·좌석수·상태.
- 개설 시 **행사 식별코드 자동 발급**(`EXPO-{연도}-{영문 대문자+숫자 6자리}`), 중복 시 재생성.
- 상태: DRAFT(작성중)/OPEN(예약중)/CLOSED(마감)/HIDDEN(숨김). 고객 목록엔 OPEN만 노출.
- 삭제는 예약이 없을 때만 허용, 있으면 CLOSED 전환 유도.

#### F-EXPO-02 콘텐츠·썸네일 편집
- 소개문·일정·가격·좌석·썸네일(URL) 수정. 박람회 관리자는 자기 박람회만.

#### F-BANNER-01 VIP 배너 등록/노출순위
- 입력: 제목·이미지 URL·링크·노출순위(priority)·노출기간(start/end)·활성.
- 고객 홈 상단에 활성·기간 내 배너를 **priority 오름차순(작은 값이 먼저)** 으로 노출. **과금·클릭추적 없음.**

### 4.3 고객 예약·결제·QR
#### F-HOME-01 행사 목록/검색·필터 + 배너
- 검색: 키워드(제목), 필터: 카테고리·날짜(기간). 정렬: **임박순(start_at 오름차순)/최신순(created_at 내림차순)**.
- 상단 VIP 배너 캐러셀. 각 카드에 남은 좌석·가격 표기.

#### F-EXPO-VIEW-01 행사 상세
- 소개·일정·장소·가격·**남은 좌석**(capacity − 확정 예약 인원 합) 표시.
- 좌석 0이면 "매진" 표기, 예약 버튼 비활성.

#### F-RSV-01 예약 신청
- 입력: 인원(headcount)·연락처·동의. 검증: 남은 좌석 ≥ 인원.
- 생성: Reservation(status=PENDING, amount=가격×인원, reservation_no 발급).

#### F-PAY-01 결제(카카오페이/Mock)
- PortOne V2(EASY_PAY, 카카오페이)로 결제창 호출 → 서버가 `paymentId`로 **검증(status=PAID·금액 일치)** 후 확정.
- PortOne 키(`PORTONE_STORE_ID/CHANNEL_KEY/API_SECRET`) 미설정 시 **Mock 결제**로 폴백.
- 성공: Payment(PAID) 기록, Reservation → **CONFIRMED**, checkin_token·QR 발급, paid_at 기록.
- **실패**: Payment(FAILED) 기록, Reservation은 **PENDING 유지**(재결제 가능). PENDING은 좌석을 점유하지 않으므로 실패건 정리 로직 불필요.
- **좌석 확정 재검증**: 결제 성공(CONFIRMED 전이) 트랜잭션 안에서 남은 좌석 ≥ 인원을 다시 확인, 부족하면 확정 거부(오버셀 방지).

#### F-TICKET-01 예약확인 + QR 티켓
- 예약번호·상태·금액·QR 표시. QR payload = `예약ID + checkin_token`.
- QR 이미지는 서버 생성(라이브러리/데이터 URI). 나의 예약에서 재조회 가능.

#### F-MYRSV-01 나의 예약/취소 요청
- 본인 예약 목록·상세·QR 재조회. CONFIRMED 건에 **취소 요청**(→ CANCEL_REQUESTED).
- 실제 환불 송금은 범위 밖(상태만 관리).

### 4.4 박람회 관리자 운영
#### F-RSV-ADM-01 예약자 명단/검색/상태변경/CSV
- 자기 박람회 예약 목록(이름·전화·인원·상태·결제금액). 검색: 이름/전화/예약번호.
- 상태 변경(예: 취소 요청 승인 → CANCELLED). **CSV 다운로드**(UTF-8 BOM, 컬럼: 예약번호·이름·전화·인원·상태·금액) 제공.
- N+1 회피: 목록 조회 시 `Customer` fetch join.

#### F-CHK-01 체크인 처리(코드/QR)
- 입력: 체크인 코드(또는 예약번호). 대조 성공 시 Reservation → **CHECKED_IN**, checked_in_at 기록.
- 이미 체크인/미확정/타 박람회 건은 거부 메시지.

### 4.5 대시보드·통계
#### F-DASH-01 전체 대시보드 (전체 관리자)
- KPI: 총 박람회 수, 총 예약 수, **총 매출(Reservation status ∈ CONFIRMED·CHECKED_IN의 amount 합)**, 평균 체크인율(= CHECKED_IN 인원 ÷ (CONFIRMED+CHECKED_IN) 인원), 박람회별 비교.
- 매출은 Payment가 아니라 **Reservation 기준**으로 집계한다(Payment는 저장소 없는 하위 엔티티라 KPI 소스로 쓰지 않음).

#### F-DASH-02 박람회 대시보드 (박람회 관리자)
- 자기 박람회: 예약 수, 결제 금액, 체크인율, 좌석 소진율.

#### F-STAT-01 통합 통계
- 기간별 예약 추이, 인기 행사(예약순) 등 집계 카드/차트.

### 4.6 LMS (교육)
#### F-LMS-01 교육 생성
- 입력: 제목·설명·**YouTube 링크**. 링크에서 `videoId` 파싱·저장. 활성 여부.

#### F-LMS-02 교육 배정
- 대상(박람회 관리자) 선택 → TrainingAssignment 생성(status=NOT_STARTED).
- 동일 대상·교육 중복 배정 방지.

#### F-LMS-03 교육 수강(시청·진행률)
- 배정된 교육 화면에서 YouTube IFrame Player로 재생.
- 클라이언트가 **5초 주기로 재생 위치(currentTime)·영상 길이(duration)** 를 서버에 전송(Ajax).
- 서버: `progress_rate = max(시청 위치) ÷ duration × 100` 갱신, 최초 재생 시 IN_PROGRESS.
- **재생 위치가 영상 길이의 95% 이상(영상 끝) 도달 시 status=COMPLETED, completed_at 기록.**
- 화면 요소(첨부 스크린샷 톤): 이수 현황(미시작/수강중/이수완료), 예상 소요시간, 이수 혜택, 영상 플레이어, 시청률 바(%).
- 검증: 본인 배정분만 접근·갱신(소유권 체크). 되감기해도 최대 도달 위치 기준 유지.

#### F-LMS-04 수강율 조회 (전체 관리자)
- 교육별 대상자 목록과 각자의 시청률(%)·이수 여부·이수일 표시.
- N+1 회피: `Training`·`AdminUser(assignee)` fetch join.

## 5. 업무 흐름 (상태 전이도)
### 5.1 예약(Reservation)
```
PENDING(결제대기)
  └─[결제 성공]→ CONFIRMED(예약확정·QR발급)
        ├─[현장 체크인]→ CHECKED_IN(입장완료)
        └─[취소 요청]→ CANCEL_REQUESTED ─[관리자 승인]→ CANCELLED
```
- 전이 검증은 service 계층. 좌석 잔여는 **CONFIRMED·CHECKED_IN 인원 합** 기준으로 계산(체크인해도 좌석은 계속 점유).

### 5.2 교육(TrainingAssignment)
```
NOT_STARTED ─[재생 시작]→ IN_PROGRESS ─[영상 끝(≈95%+) 도달]→ COMPLETED
```

## 8. 비기능 요구사항 (경량 범위)
- 비밀번호·PIN BCrypt. 소유권 검증(고객 본인/박람회 관리자 담당 박람회/수강 본인).
- 결제 검증은 서버 사이드. 폴링/Ajax 실패는 무시하고 다음 주기 재시도.
- H2 파일 모드로 재기동 후에도 데이터 유지. 시드 데이터로 즉시 시연 가능.

## 9. 제외 기능 (명세 대비)

  QR 카메라 스캐너, 다국어, 이메일/SMS 템플릿, MongoDB, React/MyBatis/CI·CD 인프라.

## 10. 확정된 정책
- LMS 수강 대상 = 박람회 관리자. 이수 기준 = 영상 끝(≈95%+) 도달.
- 결제 = 카카오페이(PortOne) + Mock 폴백. 결제 성공 시에만 예약 확정·QR 발급.
- 좌석 잔여 = capacity − CONFIRMED/CHECKED_IN 인원 합. 초과 예약 불가.
- 삭제보다 상태/비활성 우선(데이터 보존).
- **좌석은 결제 성공(CONFIRMED) 시점에만 점유**된다. PENDING(결제대기)은 좌석을 홀드하지 않으므로 미결제 예약 만료·정리 로직이 필요 없다.
- **취소 승인(CANCELLED)** 시 해당 예약은 좌석 집계에서 빠져 좌석이 자동 반환된다(환불 송금은 범위 밖, 상태만 관리).
- 매출·좌석 집계는 모두 **Reservation 기준(CONFIRMED·CHECKED_IN)** 으로 통일한다.
- LMS 이수 임계 상수 = **0.95**(영상 길이의 95%).

## 11. 입력 검증 규칙
- **전화번호**: 숫자 10~11자리(하이픈 제거 후 저장), Customer.phone 유니크.
- **PIN**: 숫자 4~6자리, BCrypt 저장. 관리자 **username**: 영숫자 4~20자, 유니크.
- **headcount**: 1 이상 & 남은 좌석 이하. **price·capacity**: 0 이상 정수.
- **일정**: `start_at < end_at` 필수. **행사 코드**: `EXPO-{연도}-{대문자영숫자 6자리}`, 충돌 시 재생성.
- **결제 금액**: `Reservation.amount = price × headcount` 를 **서버에서 재계산**(클라이언트 전달 금액 신뢰 금지).
- **YouTube 링크**: `youtu.be/{id}`·`youtube.com/watch?v={id}` 형식에서 videoId 파싱, 실패 시 등록 거부.
- 오류/거부 메시지는 한글 문안으로 노출(로그인 실패·매진·중복 체크인·타 박람회 건 등).
